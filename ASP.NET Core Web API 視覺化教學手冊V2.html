4. Model 實體模型配置 (C# Classes)

Employee.cs 配置

using System.ComponentModel.DataAnnotations;

public class Employee
{
    [Key]
    public int EmployeeId { get; set; }
    [Required]
    [StringLength(100)]
    public string EmployeeName { get; set; } = "";
    [Required]
    public string Department { get; set; } = "";
    public DateTime DateOfJoining { get; set; }
    public string PhotoFileName { get; set; } = "";
}

為什麼 Model 很重要？

  • 作為 DTO：接收來自前端的 JSON 資料並自動轉型。
  • 作為 Entity：對應資料庫表格欄位，與 Entity Framework 配合。

5. Interface (介面) 使用與說明

I IEmployeeService.cs 

public interface IEmployeeService
{
    IEnumerable<Employee> GetAll();
    Employee AddEmployee(Employee emp);
}

介面作為契約

Controller 只依賴介面，系統透過 Program.cs 注入實作物件。這就是 依賴注入 (DI)。

6. Repository (倉儲層) 使用與說明

Repository 負責與資料庫溝通（例如下 SQL 或使用 DbContext）。這可以將「如何拿資料」與「如何處理資料」分開。

R EmployeeRepository.cs 

public class EmployeeRepository : IEmployeeRepository
{
    // 這裡通常會注入 DbContext
    public Employee Insert(Employee emp)
    {
        // 實作：_context.Employees.Add(emp);
        // _context.SaveChanges();
        return emp;
    }
}

為什麼需要 Repository？

  • 1. 隔離關注點： Service 只專注於業務判斷，而 Repository 只專注於資料持久化 (CRUD)。
  • 2. 降低重複： 相同的資料存取邏輯 (如：根據員工名稱搜尋) 可以在多個 Service 間共用。

7. CRUD 流程詳解：路由匹配到資料庫存取

以「新增一位員工 (POST)」為例，觀察 Model、Service、Repository 的交互：

步驟 A：客戶端發出請求

{ "EmployeeName": "李大明", "Department": "IT", ... }

步驟 B：路由引擎匹配 (Routing Match)

URL: /api/Employees + Verb: POST => 執行 EmployeesController.Post()

步驟 C：Controller 接收並呼叫 Service

Controller 透過 IEmployeeService 進入業務流程。

[HttpPost]
public IActionResult Post(Employee emp) 
{
    var result = _service.AddEmployee(emp); // 傳遞 Model 給 Service
    return CreatedAtAction(nameof(Get), new { id = result.EmployeeId }, result);
}

步驟 D：Service 執行業務驗證與調度

Service 驗證部門是否存在後，將 Model 交給 Repository 儲存。

public Employee AddEmployee(Employee emp) 
{
    if (string.IsNullOrEmpty(emp.EmployeeName)) throw new Exception();
    return _repository.Insert(emp); // 將資料持久化職責交給 Repository
}

步驟 E：Repository 執行資料庫存取

INSERT INTO [Employees] (...) VALUES (...);
-- Repository 使用 EF Core 或 Dapper 執行實際 SQL。

步驟 F：回復客戶端

API 回傳 HTTP 201 狀態碼與 JSON 結果，完成整個請求流程。
