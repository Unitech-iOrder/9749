ASP.NET Core Web API 實戰教學

從基礎概念到 Service Layer 分層架構

1. 請求的生命週期 (Life of a Request)

當客戶端發送一個請求到 Web API 時，它會經過以下處理管線：

客戶端
→
Step 1
Middleware
驗證、日誌
→
Step 2
Routing
路徑匹配
→
Step 3
Controller
處理請求
→
Step 4
Service
業務邏輯
→
資料庫

完成處理後，資料會依照原路徑封裝為 HTTP Response 回傳給客戶端。

2. 分層架構實作 (CRUD)

1 Service Layer (介面與實作)

我們定義 IProductService 來規範功能，這使得 Controller 不需要知道底層是如何讀取資料的。

IProductService.cs
public interface IProductService 
{
    IEnumerable<Product> GetAll();
    Product Add(Product item);
}
ProductService.cs
public class ProductService : IProductService 
{
    private readonly List<Product> _db = new();

    public IEnumerable<Product> GetAll() => _db;

    public Product Add(Product item) 
    {
        _db.Add(item);
        return item;
    }
}

2 依賴注入 (Dependency Injection)

在 Program.cs 註冊服務，這就是將「介面」與「實作」連結在一起的關鍵。

// Program.cs
var builder = WebApplication.CreateBuilder(args);

// 當有人要求 IProductService，請給他 ProductService 實例
builder.Services.AddScoped<IProductService, ProductService>();

3 Controller (門面)

Controller 透過建構子注入獲得 Service，只負責調度，不處理邏輯。

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase 
{
    private readonly IProductService _service;

    public ProductsController(IProductService service) => _service = service;

    [HttpGet]
    public IActionResult Get() => Ok(_service.GetAll());
}

學習總結

  • 解耦合：Controller 不再依賴具體實作，測試更容易。
  • 單一職責：Service 專注邏輯，Controller 專注傳輸。
  • 擴充性：未來若要更換為 Oracle 或 MSSQL 資料庫，只需建立新的 Service 實作並修改 DI 註冊。
